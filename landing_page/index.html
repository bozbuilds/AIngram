AIngram Pro

Scaling to millions of memories with enterprise features.

GPU-resident neural caching

Your vector database gets slower as it gets larger. That's a law of physics, not a software bug. Every query touches more data. The only way to break this is to keep the hot memories somewhere that doesn't scale linearly with the total database size. That's what the Pro memory cache does.

CRDTs for Agent Memory

Multi-agent systems have a coordination problem that single-agent systems don't: when Agent A and Agent B both update the same memory, whose version wins? The naive answer is 'last write wins.' The correct answer is 'both versions are preserved, and you can query the merge history.' That's what distributed systems people have been doing with CRDTs for a decade. We applied it to agent memory.

Asymmetric Consolidation

Human memory doesn't store everything equally. Novel information that violates expectations is retained longer. Repeated patterns that confirm what you already know fade. There's a reason you remember the one time your code worked in a surprising way better than the hundred times it worked as expected. We built this asymmetry into the Pro consolidation system: memories that contain high-surprise findings are scheduled for reinforcement more aggressively than memories that are predictable.

Learned Search Index

A 768-dimensional float32 vector takes 3KB of storage and 3KB of memory bandwidth on every comparison. At 1 million memories, that's 3GB for the index alone, and every search scans all of it. The Pro search index learns which parts of the database are relevant to which query types. The result is ~200x faster search at 1 million vectors, versus a standard vector search.

Causal Graph Explicitly Tracks Causality

Vector search finds memories that are semantically similar to your query. That's useful. But similarity isn't causality. "The model improved" and "we reduced the learning rate" might appear in the same memory, but the causal direction matters: did reducing the learning rate cause the improvement, or did both happen to coincide? The Pro causal graph tracks this distinction explicitly. After 50 experiments, your agent doesn't just know what happened — it has a model of why.

Frequently asked questions

What is AIngram?

AIngram is an open-source Python library that gives AI agents persistent, searchable memory backed by a local SQLite file. It combines FTS5 keyword search, dense vector search, and knowledge graph traversal fused via Reciprocal Rank Fusion — achieving 95.5% recall@10 on LongMemEval-S. No cloud service, no managed infrastructure, no API keys required.

Does AIngram require a GPU or cloud service?

No. AIngram runs entirely on your local machine. Embeddings use a quantized ONNX model (Nomic MiniLM, 768-dim) that runs on CPU. The database is a single portable .db file — no server, no Docker, no cloud account. A GPU is optional and only relevant when using the aingram-AR autoresearch scaffolding in swarm mode.

How does AIngram compare to Pinecone, Chroma, or Weaviate?

Managed vector databases require cloud infrastructure, API keys, and ongoing costs. AIngram stores everything in a single SQLite file on disk, optimized for agent memory workloads (thousands to low millions of entries). You get zero remote latency, tamper-evident Ed25519 signing, full-text search, and a knowledge graph — all with zero deployment overhead. AIngram Pro extends this to ~1M entries with GPU-resident caching at 200x the speed of standard vector search.

What AI frameworks and tools does AIngram integrate with?

AIngram ships with native adapters for LangChain, CrewAI, LangGraph, AutoGen, and smolagents. It also includes a built-in MCP server compatible with Claude Desktop, Cursor, and Cline — allowing any MCP-compatible AI tool to call remember and recall without writing Python. Install with pip install aingram and drop it into your existing agent stack.

How accurate is AIngram's memory recall?

AIngram achieves 95.5% recall@10 on LongMemEval-S, the most rigorous publicly available AI memory benchmark, with a median query latency of 22ms. The hybrid retrieval pipeline — FTS5 + dense vector search + knowledge graph traversal via Reciprocal Rank Fusion — consistently outperforms any single retrieval method on multi-hop and temporally complex queries.
